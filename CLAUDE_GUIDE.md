# Эффективная работа с Claude Code CLI

## Принцип №1: Чем конкретнее задача — тем лучше результат

Плохо:
```
добавь SignalR
```

Хорошо:
```
Добавь SignalR Hub с именем ChatHub в SignalrExample/Hubs/ChatHub.cs.
Hub должен иметь метод SendMessage(string user, string message),
который рассылает сообщение всем подключённым клиентам.
Зарегистрируй Hub в Program.cs по маршруту /chat.
```

**Правило:** укажи файл, класс, метод, поведение и ограничения.

---

## Слэш-команды, которые реально экономят время

| Команда | Когда использовать |
|---|---|
| `/plan` | Перед крупной задачей — согласуй подход ДО того, как Claude начнёт писать код |
| `/code-review` | После внесения изменений — проверка на баги и упрощения |
| `/code-review ultra` | Глубокое ревью текущей ветки через облачных агентов |
| `/run` | Запустить проект и убедиться, что изменение работает |
| `/verify` | Проверить конкретное поведение в живом приложении |
| `/simplify` | Упростить код после реализации фичи |
| `/security-review` | Перед мержем — проверка уязвимостей |
| `/init` | Создать/обновить CLAUDE.md для нового репозитория |
| `/context` | Посмотреть сколько токенов использовано |
| `/clear` | Сбросить контекст сессии (начать с чистого листа) |

## Режим планирования (`/plan`) — обязателен для крупных задач

Для задач, затрагивающих 3+ файла или требующих архитектурных решений:

1. Открой план-режим: `/plan`
2. Опиши задачу — Claude предложит подход и список изменений
3. Уточни, скорректируй, задай вопросы
4. Когда план устраивает — подтверди, Claude перейдёт к реализации

Это предотвращает ситуацию, когда Claude делает 200 строк кода не в том направлении.

## Как давать контекст

### Ссылайся на файлы явно
```
В файле SignalrExample/Program.cs добавь...
```

Не надейся, что Claude угадает нужный файл из десятков похожих.

### Давай примеры желаемого поведения
```
Метод должен возвращать DTO такого вида:
{ "user": "Иван", "message": "Привет", "timestamp": "2026-06-09T10:00:00" }
```

### Указывай ограничения
```
Не меняй сигнатуру существующего метода Get() в WeatherForecastController.
Используй только уже подключённые NuGet-пакеты.
```

## Итеративная работа — лучшая стратегия

Вместо одного большого запроса — серия маленьких:

```
1. Сначала: "Добавь модель ChatMessage с полями User и Message"
2. Потом:   "Теперь создай ChatHub с методом SendMessage"  
3. Затем:   "Зарегистрируй Hub в Program.cs"
4. Потом:   "Напиши простой HTML-клиент для проверки"
```

После каждого шага можно запустить `/verify` или `/code-review`.

## Работа с Git через Claude

Claude понимает git-контекст и умеет:

```
# Попросить создать коммит с осмысленным сообщением
"Создай коммит с текущими изменениями"

# Создать PR
"Создай pull request в main с описанием изменений"

# Ревью ветки
"/code-review ultra" — анализирует всю ветку относительно main
```

**Важно:** Claude попросит подтверждение перед push и деструктивными операциями.

---

## Символ `!` для выполнения команд в терминале

В промпте можно запустить команду напрямую:

```
! dotnet build
! dotnet test
! git log --oneline -10
```

Вывод попадёт в контекст разговора — Claude увидит ошибки и отреагирует.

## Управление контекстом

- **`/context`** — смотри процент заполнения (70%+ = пора думать о `/clear`)
- **`/clear`** — сбрасывает историю диалога, но Claude.md и файлы остаются
- После `/clear` кратко напомни что делаешь: *"Продолжаем добавлять SignalR. Уже есть ChatHub, нужно добавить авторизацию"*

## Типичные шаблоны запросов

### Исправление бага
```
В методе [имя метода] в [файл:строка] происходит [описание проблемы].
Ожидаемое поведение: [что должно быть].
Фактическое поведение: [что происходит].
Не трогай [что не надо трогать].
```

### Новая фича
```
Добавь [фичу] в [место].
Она должна делать [поведение].
Входные данные: [что принимает].
Выходные данные: [что возвращает].
Не добавляй ничего сверх этого.
```

### Рефакторинг
```
Рефактори [файл/метод] для [цели: читаемость/производительность/тестируемость].
Сохрани публичный API без изменений.
```

### Объяснение кода
```
Объясни как работает [файл/класс/метод].
Сфокусируйся на [конкретном аспекте].
```

## Память между сессиями

Claude сохраняет факты о проекте между сессиями в папке `.claude/projects/`.

Чтобы зафиксировать важное решение:
```
"Запомни: мы используем только PostgreSQL через Dapper, без EF Core"
"Запомни: порт приложения 5555, не менять"
```

---

## Что Claude делает автоматически (не надо просить)

- Читает нужные файлы перед редактированием
- Сохраняет стиль кода проекта (отступы, соглашения об именовании)
- Спрашивает подтверждение перед push, удалением файлов, force-операциями
- Параллельно выполняет независимые операции для скорости

---

## Красные флаги — когда что-то идёт не так

- Claude добавляет много кода, который ты не просил → скажи *"стоп, только то что я просил"*
- Claude не уверен в архитектуре → используй `/plan` перед продолжением
- Ответы становятся длинными и общими → `/clear` + более конкретный запрос
- Claude предлагает установить новые пакеты → уточни, нужно ли это
